Three Architectural Approaches

How should we build the rendering and layout engine?

A

Canvas-First (HTML5 Canvas / WebGL)

Render everything on a <canvas> element. Custom rendering for nodes, edges, text. Like how Excalidraw works under the hood.

Pros

  • Unlimited freeform drawing — no DOM constraints
  • Great performance with thousands of nodes
  • Full control over visual rendering

Cons

  • Text rendering is hard — no native HTML text flow, wrapping, or rich formatting
  • Accessibility is poor — screen readers can't see canvas content
  • Huge implementation effort — must build text layout, hit detection, scrolling from scratch
  • Bad fit for text-heavy diagrams
B

DOM-First with SVG Edges (Recommended)

Nodes are React components positioned via CSS transforms on an infinite pannable/zoomable canvas. Edges are SVG paths rendered in an overlay. Auto-layout powered by dagre or ELK.

Pros

  • Native HTML text — rich formatting, wrapping, and flexible content for free
  • React component model — nodes are just components, easy to customize
  • Accessible out of the box — standard DOM elements
  • Moderate implementation effort — leveraging browser layout engine
  • SVG edges give smooth curves and arrowheads

Cons

  • Performance ceiling around 500-1000 nodes (fine for decision charts)
  • Less freeform than pure canvas — but edit mode drag-and-drop covers this
C

Hybrid (DOM Nodes + Canvas Background)

React DOM nodes for content, but a canvas layer underneath for decorative elements, grid, and potentially edges. Combines approaches A and B.

Pros

  • Best of both worlds in theory
  • Canvas for visual flair, DOM for content

Cons

  • Synchronizing two rendering layers adds complexity
  • More code to maintain for marginal benefit over pure DOM+SVG
  • Overkill for decision charts that don't need decorative canvas elements
